Docker
×

Technology delivery has evolved over time; from traditional bare-metal self-maintained infrastructures to full micro-services functions - Docker is an important stepon that journey

  • Bare-Metal Legacy Applications built andmaintained on numerous servers, either self hosted or MSP
  • Tiered Legacy Applications split apart by function so front end delivery, application and data layers can be provisioned at appropriate scale
  • Stateful Virtualisation Transition from bare-metal tier delivery tovirtualised machines with software defined routes and layering
  • Stateful Containerisation Evolution from virtualisation where the containers can be flexibily built with minimal layers and scaled depending on use
  • Stateless Containerisation Utilising ESB or messaging and transaction state, individual API functions can be deployed in images and scaled on calls per function allowing a highly scalableand resilent infrastructure
  • Serverless Further abstraction taking single APUI functions or parts of API functions into single action calls that are infinitly scalable on cloud services

Docker also allows us to spin up any configuration in a custom build - or more frequently to utilise provider or community images allowing rapid proof of concept for true devops "fail-fast" culture

Microsoft AKS
×

Azure Kubernetes Service (AKS) makes it simple to deploy a managed Kubernetes cluster in Azure. AKS reduces the complexity and operational overhead of managing Kubernetes by offloading much of that responsibility to Azure. As a hosted Kubernetes service, Azure handles critical tasks like health monitoring and maintenance for you. The Kubernetes masters are managed by Azure. You only manage and maintain the agent nodes. As a managed Kubernetes service, AKS is free - you only pay for the agent nodes within your clusters, not for the masters.

Azure Policy
×

Azure Policy evaluates resources in Azure by comparing the properties of those resources to business rules(policy definitions) and combined into Policy Sets

Pooicy Sets can be assigned to any resource model that Azure supports, such as management groups, subscriptions, resource groups, or individual resources, allowing control to be applied or subdued depending on the environment.

Resources are evaluated at specific times during the resource lifecycle, the policy assignment lifecycle, and for regular ongoing compliance evaluation. The following are the times or events that cause a resource to be evaluated:

  • A resource is created, updated, or deleted in a scope with a policy assignment.
  • A policy or initiative is newly assigned to a scope.
  • A policy or initiative already assigned to a scope is updated.
  • During the standard compliance evaluation cycle, which occurs once every 24 hours.
This allows easy translation from existing business security or compliance processes and documents to automated and integrated, ultimately risk reducing enforcement or alerting rules
